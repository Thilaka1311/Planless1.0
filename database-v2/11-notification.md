# Notifications Domain

## Purpose

The Notifications domain records important events that require a user's attention.

Notifications are displayed in the in-app notification center and may also be delivered as push notifications for supported devices in the future.

The Notifications domain acts as the central inbox for all important user activity across Planless.

---

# Responsibilities

The Notifications domain is responsible for:

* Recording user notifications
* Tracking read and unread status
* Powering the in-app notification center
* Supporting future push notifications

The Notifications domain is **not** responsible for:

* Plans
* Friendships
* Wallet
* Memories

These domains simply create notifications when appropriate.

---

# Design Principles

1. Every notification belongs to exactly one user.
2. Notifications are immutable after creation.
3. Notifications may be marked as read.
4. Every notification originates from a system event.
5. Only important events generate notifications.
6. Push notifications are an optional delivery mechanism.

---

# Table

## notifications

| Column     | Type        | Description                                         |
| ---------- | ----------- | --------------------------------------------------- |
| id         | UUID        | Primary key                                         |
| user_id    | UUID        | References `users.id`                               |
| type       | ENUM        | Notification type                                   |
| title      | TEXT        | Notification title                                  |
| body       | TEXT        | Notification message                                |
| data       | JSONB       | Optional metadata (e.g. `plan_id`, `friendship_id`) |
| is_read    | BOOLEAN     | Read status                                         |
| created_at | TIMESTAMPTZ | Notification creation timestamp                     |
| read_at    | TIMESTAMPTZ | Timestamp when notification was read (nullable)     |

# Relationships

Each notification:

* Belongs to one user.
* Is generated by one system event.
* May reference another entity through its metadata.

Examples:

* Plan invitation
* Friend request
* Payment received
* Memory generated

---

# Workflow

1. A system event occurs.
2. The responsible domain creates a notification.
3. The notification appears in the user's notification center.
4. If push notifications are enabled, the notification may also be delivered to the user's device.
5. When opened, the notification is marked as read.

---

# Constraints

* Every notification belongs to exactly one user.
* Notifications cannot be reassigned.
* `read_at` is populated only when `is_read` becomes `true`.

---

# User Experience

The notification center displays notifications in reverse chronological order.

Unread notifications are highlighted.

Selecting a notification navigates the user directly to the relevant content.

Examples:

* Plan invitation → Opens the invitation.
* Friend request → Opens the sender's profile.
* Payment received → Opens the wallet.
* Memory generated → Opens the memory.

---

# Future Push Notifications

Initially, notifications are displayed only within the application.

Future versions may deliver the same notifications as browser push notifications using device subscriptions.

This enhancement does not require changes to the notification schema.

---

# Design Philosophy

Notifications are not the source of truth.

They simply inform users that something important has happened elsewhere in the system.

Plans, Friendships, Wallet, and Memories own their respective business logic.

The Notifications domain provides a single, consistent way of informing users about important events while avoiding unnecessary interruptions.
