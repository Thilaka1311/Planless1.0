# MEMORIES LIFECYCLE

## Purpose

Memories represent the outcome of a real-world experience.

A Memory exists because a Plan happened.

Memories are the final stage of the Planless core loop:

Discovery

↓

Plan

↓

Experience

↓

Memory

A Memory should answer:

"What happened?"

not

"What content was posted?"

---

# Core Philosophy

Memories should feel:

* personal
* social
* reflective
* lightweight
* authentic

Memories are not social media posts.

Memories are not content feeds.

Memories are records of real-world experiences.

---

# Memory Creation Flow

A Memory can only be created after a plan is completed.

Trigger:

Plan Status

ACTIVE

↓

COMPLETED

Result:

Memory becomes available.

Participants may contribute memory information.

---

# Memory Ownership

Memories belong to:

* the Plan
* the Participants

Memories are attached to the plan that generated them.

# Football Memory Flow

Trigger:

Football Plan

↓

COMPLETED

Host may record match results.

---

## Required

### Final Score

Example:

Team A 4

Team B 2

The final score becomes part of the memory.

---

## Optional

### Goal Scorers

Host may record:

* Player
* Goals scored

Example:

John - 2 Goals

Alex - 1 Goal

---

### MVP

Host may select:

Most Valuable Player

Optional.

---

# Football Memory Rules

The host is responsible for recording the final result.

Result data should remain editable until saved.

Once saved:

Memory becomes historical.

---

# Badminton Memory Flow

Trigger:

Badminton Plan

↓

COMPLETED

Badminton supports team rotation.

---

# Team Creation

Host may create teams during the plan.

Teams may be changed live.

Teams are not fixed.

---

# Match Completion

After each match:

Host may enter:

Final Score

Example:

Team A 21

Team B 17

---

# Team Rotation

After a score is entered:

Host may immediately:

Switch Teams

↓

Create Next Match

↓

Enter Next Result

This process may repeat throughout the session.

---

# Badminton Memory Rules

Badminton memories should support multiple match results within a single plan.

The host controls score entry.

The system should remain lightweight and fast.

---

Movie Plan

# Rating

Rating Type:

Slider

Range:

1 → 10

# Movie Memory Rules

Ratings are optional.

The experience matters more than the score.

---

# Dining Memory Flow

↓

# Rating

Rating Type:

Slider

Range:

1 → 10

Examples:

10/10

---

# Dining Memory Rules

Ratings are optional.

---

# Visibility Rules

Participants may view memories for plans they participated in.

Hosts may view memories for plans they hosted.

Users should not gain access to unrelated memories.

---

# Historical Preservation Rules

Memories should remain permanently attached to the plan.

Deleting a Circle should not remove memories.

Leaving a Circle should not remove memories.

Removing friendships should not remove memories.

Historical experiences should remain intact.

---

# Profile Integration

Memories may appear on user profiles.

Memory visibility should follow participation permissions.

The profile should reflect real-world experiences.

---

# Database Rules

The database is the source of truth.

Memory records should be attached to:

Plan

↓

Memory

Future memory types may expand over time.

# Non-Negotiable Rules

1. Memories can only be created after a plan is completed.

2. Memories are attached to plans.

3. Memories are not social media posts.

4. Memories are not content feeds.

5. Football memories support final scores.

6. Football memories may support goal scorers.

7. Football memories may support MVP selection.

8. Badminton memories support multiple match results.

9. Movie memories support ratings.

10. Dining memories support ratings.

11. Database is the source of truth.

12. Historical memories must never be deleted automatically.

13. Memories should always reinforce:

Discovery

↓

Plan

↓

Experience

↓

Memory
